# 第1章 サンプルアプリケーションの構築

第0章で環境要件とNamespace設計を確認した。本章では、本書全体を通して使用するサンプルアプリケーションをKubernetes上に構築する。3〜4サービスで構成されるマイクロサービスアプリケーションをデプロイし、Kustomizeによる環境差分の管理を学ぶ。そして最後に、この「素のKubernetes」環境が抱える課題を明示し、以降のPartで解決すべきテーマを整理する。

### なぜマイクロサービスアーキテクチャを採用するのか

本書のサンプルアプリケーションは、意図的にマイクロサービスアーキテクチャを採用している。商品管理と注文管理を別サービスに分離することで、以下の学習効果が得られる。

- **サービス間通信の可視化**: サービス間のHTTP通信が存在するため、分散トレーシング（第4章）やService Mesh（第6〜7章）の効果を実感できる
- **独立したデプロイ**: 各サービスを個別にデプロイ・スケーリングできるため、Canaryデプロイ（第14章）の対象として適している
- **障害の伝播**: あるサービスの障害が他のサービスに影響を与えるシナリオを再現でき、Observability（Part 1）の重要性を体感できる

サービス数を3つ（フロントエンドを除く）に抑えたのは、学習のオーバーヘッドを最小限にしつつ、マイクロサービス特有の課題を十分に体験できるバランスを意図したためである。

### 技術スタックの選定理由

バックエンドサービスにPython（FastAPI）を採用した理由は以下の通りである。

- **FastAPI**: 型ヒントに基づく自動バリデーションとOpenAPIドキュメント自動生成を備え、少ないコード量でRESTful APIを構築できる。非同期処理（async/await）にも対応しており、高いパフォーマンスを発揮する
- **uvicorn**: ASGI（Asynchronous Server Gateway Interface）サーバーとして、FastAPIアプリケーションをプロダクション品質で実行する
- **psycopg2**: PostgreSQLへの接続にはpsycopg2（Python用PostgreSQLアダプタ）を使用する
- **prometheus-client**: Prometheusメトリクスのエクスポートに使用する（第2章で詳述）
- **opentelemetry SDK**: 分散トレーシングの計装に使用する（第4章で詳述）
- **python-json-logger**: 構造化ログ（JSON形式）の出力に使用する（第3章で詳述）

各サービスのDockerfileは `python:3.12-slim` をベースイメージとし、`requirements.txt` で依存パッケージを管理する。

### サービス間通信フロー

ユーザーからのリクエストは以下の流れで処理される。

1. ユーザーがブラウザからフロントエンドにアクセスする
2. フロントエンドがAPIゲートウェイにHTTPリクエストを送信する
3. APIゲートウェイがリクエストのパスに基づいて商品サービスまたは注文サービスにルーティングする
4. 各バックエンドサービスがPostgreSQLにデータを読み書きする
5. レスポンスが逆順に返される

リスト1.1に、APIゲートウェイのルーティング処理の抜粋を示す。

```python
# リスト1.1: APIゲートウェイの main.py（抜粋）
import os
import httpx
from fastapi import FastAPI, Request, Response

app = FastAPI(title="API Gateway")

PRODUCT_SERVICE_URL = os.getenv(
    "PRODUCT_SERVICE_URL",
    "http://product-service:8081"
)
ORDER_SERVICE_URL = os.getenv(
    "ORDER_SERVICE_URL",
    "http://order-service:8082"
)

# /api/products → 商品サービスへプロキシ
@app.api_route("/api/products/{path:path}", methods=["GET", "POST", "PUT", "DELETE"])
async def proxy_products(request: Request, path: str = ""):
    async with httpx.AsyncClient() as client:
        url = f"{PRODUCT_SERVICE_URL}/api/products/{path}"
        response = await client.request(
            method=request.method,
            url=url,
            headers=dict(request.headers),
            content=await request.body(),
        )
        return Response(
            content=response.content,
            status_code=response.status_code,
            headers=dict(response.headers),
        )

# /api/orders → 注文サービスへプロキシ
@app.api_route("/api/orders/{path:path}", methods=["GET", "POST", "PUT", "DELETE"])
async def proxy_orders(request: Request, path: str = ""):
    async with httpx.AsyncClient() as client:
        url = f"{ORDER_SERVICE_URL}/api/orders/{path}"
        response = await client.request(
            method=request.method,
            url=url,
            headers=dict(request.headers),
            content=await request.body(),
        )
        return Response(
            content=response.content,
            status_code=response.status_code,
            headers=dict(response.headers),
        )

if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8080)
```

APIゲートウェイはリバースプロキシとして動作し、URLパスに基づいてリクエストを適切なバックエンドサービスに転送する。FastAPIの `api_route` デコレータで複数のHTTPメソッド（GET、POST、PUT、DELETE）を一括でルーティングし、`httpx.AsyncClient` で非同期にバックエンドサービスへリクエストを転送する。環境変数 `PRODUCT_SERVICE_URL` と `ORDER_SERVICE_URL` でバックエンドサービスのエンドポイントを外部化しており、これらの値はConfigMap（1.2節で解説）から注入される。サービス間の通信はすべてHTTP/RESTで行う。

### ConfigMap

リスト1.4にConfigMapマニフェストを示す。サービス間のエンドポイントやDB接続情報を外部化する。

```yaml
# リスト1.4: ConfigMap マニフェスト
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: book-app
data:
  PRODUCT_SERVICE_URL: "http://product-service.book-app.svc.cluster.local:8081"
  ORDER_SERVICE_URL: "http://order-service.book-app.svc.cluster.local:8082"
  DATABASE_HOST: "postgres.book-app.svc.cluster.local"
  DATABASE_PORT: "5432"
  DATABASE_NAME: "bookapp"
  DATABASE_USER: "bookapp"
```

#### ConfigMapで環境変数を外部化する理由

設定値をアプリケーションコードにハードコードせず、ConfigMapとして外部化する設計には以下の利点がある。

1. **環境ごとの切り替え**: 開発環境・ステージング環境・本番環境で異なるデータベースホストやサービスURLを、コードを変更せずに切り替えられる
2. **Twelve-Factor App準拠**: Twelve-Factor Appの第3原則「設定を環境変数に格納する」に従い、設定をデプロイメントの関心事として分離する
3. **Kustomize overlayとの連携**: 環境ごとのoverlayでConfigMapをパッチすることで、base側のDeploymentマニフェストを変更する必要がなくなる

#### 環境変数の設計方針

サービスURLには `<サービス名>.book-app.svc.cluster.local:<ポート>` というKubernetesの完全修飾ドメイン名（FQDN）を使用している。同一Namespace内であれば `product-service:8081` のように短縮名でも名前解決が可能であるが、以下の理由からFQDNを推奨する。

- **明示性**: どのNamespaceのどのServiceを参照しているかが一目で分かる
- **クロスNamespace通信**: 将来的にサービスを別のNamespaceに移動しても、ConfigMapの値を変更するだけで対応できる
- **DNS解決の効率**: FQDNを使用すると、KubernetesのDNS解決でサーチドメインの順次試行をスキップでき、わずかながらレイテンシが改善される

> **注意**: 本番環境ではデータベースのパスワードはSecretリソースで管理すべきである。本書ではハンズオンの簡略化のためConfigMapを使用する。Secretリソースの詳細はPart 3（第9章）で解説する。

## 1.3 Kustomizeの基本 ― base + overlays

### なぜKustomizeが必要か

前節で作成したマニフェストは、OKE環境を前提としている。しかし、kind環境ではIngressコントローラが異なり、StorageClassも変わる。環境ごとにマニフェストをコピーして修正するのは、保守性の観点から望ましくない。マニフェストのコピーは「設定のドリフト」を引き起こす。baseに変更を加えたとき、すべてのコピーに同じ変更を手動で反映する必要があり、漏れが発生しやすい。

Kustomize（カスタマイズ）は、Kubernetesマニフェストをテンプレートなしでカスタマイズする仕組みである。Helmのようなテンプレートエンジンとは異なり、有効なYAMLファイルをそのまま使用する。そのため、baseのマニフェストを直接 `kubectl apply` できるという利点がある。baseディレクトリに環境非依存の共通マニフェストを配置し、overlaysディレクトリで環境固有の差分のみを定義する。図1.3にこの構造を示す。

図1.3: Kustomizeのbase + overlays構造図

```mermaid

    Base[k8s/base/<br/>共通マニフェスト<br/>deployment.yaml<br/>service.yaml<br/>configmap.yaml<br/>ingress.yaml] --> OKE[k8s/overlays/oke/<br/>OCI Load Balancer<br/>OCI Block Volume CSI<br/>OCIR]
    Base --> Kind[k8s/overlays/kind/<br/>NGINX Ingress + NodePort<br/>hostPath<br/>ローカルレジストリ]
    Base --> Custom[k8s/overlays/xxx/<br/>読者独自の環境]

    style Custom fill:#9E9E9E,color:#fff
```

### 環境間で差異が出るポイント

OKE環境とkind環境で主に差異が出るのは以下の3点である。

| 項目 | OKE | kind |
|------|-----|------|
| Ingress | OCI Load Balancer | NGINX Ingress + NodePort |
| StorageClass | OCI Block Volume CSI | hostPath |
| レジストリ | OCIR | ローカルレジストリ（localhost:5001） |

### Kustomize overlayのパッチ方式

overlayではpatchesフィールドにパッチファイルを指定する。パッチには2つの方式がある。

1. **Strategic Merge Patch**: 変更したいフィールドだけを記述する。既存のマニフェストとマージされる。リスト1.7、1.8で使用しているのはこの方式である
2. **JSON Patch**: RFC 6902に準拠した操作（add、remove、replace等）を明示的に指定する。フィールドの削除や配列の特定要素の変更に適している

本書では、読みやすさを重視してStrategic Merge Patchを主に使用する。

また、overlayのkustomization.yamlで `images` フィールドを使用している点に注目する。この機能により、base側のDeploymentマニフェストを一切変更せずに、コンテナイメージのレジストリを環境ごとに切り替えられる。OKE環境ではOCIR（Oracle Cloud Infrastructure Registry）を、kind環境ではローカルレジストリ（localhost:5001）を使用する。

### kustomize buildの実行

以下のコマンドで、最終的なマニフェストを生成できる。

```bash
# OKE環境向けマニフェストの生成と確認
$ kustomize build k8s/overlays/oke/

# 直接適用する場合
$ kubectl apply -k k8s/overlays/oke/

# 生成結果をファイルに保存して確認する場合
$ kustomize build k8s/overlays/oke/ > /tmp/rendered.yaml
$ less /tmp/rendered.yaml
```

`kustomize build` は最終マニフェストを標準出力に出力する。`kubectl apply -k` は `kustomize build` の結果を直接適用する短縮形である。初めてKustomizeを使用する場合は、`kustomize build` の出力を目視で確認してから `kubectl apply` を実行することを推奨する。

> **補足**: `kubectl apply -k` で使用されるKustomizeのバージョンは、kubectlに組み込まれたものである。スタンドアロン版の `kustomize` コマンドとはバージョンが異なる場合がある。本書ではスタンドアロン版のKustomize v5.xを前提としている。

### Kustomizeのトラブルシューティング

Kustomizeの使用時に遭遇しやすい問題と対処方法を以下にまとめる。

**問題1: `error: no matches for OriginalId` が発生する**

overlayのパッチで指定したリソースがbase側に存在しない場合に発生する。パッチファイルの `metadata.name` と `kind` がbase側のリソースと一致しているか確認する。

```bash
# base側のリソース一覧を確認
$ kustomize build k8s/base/ | grep -E "^kind:|^  name:"
```

**問題2: `accumulating resources` エラーが発生する**

kustomization.yamlの `resources` に指定したファイルが存在しない場合に発生する。ファイルパスの綴りやディレクトリ構造を確認する。

**問題3: overlayで変更が反映されない**

パッチファイルの `apiVersion`、`kind`、`metadata.name` がbase側のリソースと完全に一致している必要がある。特にNamespaceを指定している場合、パッチ側でも同じNamespaceを指定する必要がある。

```bash
# 差分を確認するため、baseとoverlayの出力を比較する
$ diff <(kustomize build k8s/base/) <(kustomize build k8s/overlays/oke/)
```

### 読者独自のoverlayを作成する

読者が独自の環境用overlayを作成する場合、以下の手順で行う。

1. `k8s/overlays/<環境名>/` ディレクトリを作成する
2. kustomization.yamlを作成し、`../../base` をresourcesに指定する
3. 環境固有のIngress、StorageClass、レジストリ設定をパッチとして定義する
4. `kustomize build` で生成結果を確認する
5. `kubectl apply --dry-run=client -k k8s/overlays/<環境名>/` でドライランを実行し、エラーがないことを確認する

### デプロイ

リスト1.9にデプロイ手順を示す。

```bash

# OKE環境の場合
```

### デプロイ時のトラブルシューティング

デプロイ時に遭遇しやすい問題と対処方法を以下にまとめる。

**Podが `Pending` のまま変化しない場合**:
- ノードのリソース（CPU、メモリ）が不足している可能性がある。`kubectl describe pod` の Events欄で `Insufficient cpu` や `Insufficient memory` を確認する
- PersistentVolumeClaimが `Pending` の場合は、StorageClassが正しく設定されているか確認する

**Podが `ImagePullBackOff` になる場合**:
- コンテナイメージ名とタグが正しいか確認する
- プライベートレジストリを使用している場合は、ImagePullSecretが設定されているか確認する
- OKE環境でOCIRを使用する場合は、`kubectl create secret docker-registry` でレジストリ認証情報を事前に作成する必要がある

**Podが `CrashLoopBackOff` になる場合**:
- `kubectl logs <pod名> -n book-app --previous` でクラッシュ前のログを確認する
- ConfigMapの環境変数（DATABASE_HOST等）が正しいか確認する。特にDNS名の綴り間違いに注意する
- PostgreSQL Podが起動完了する前にバックエンドサービスが接続を試みてクラッシュすることがある。この場合、Podの再起動で自動的に復旧する

**ServiceのEndpointsが空の場合**:
- Serviceの `selector` とDeploymentの `labels` が一致しているか確認する

```bash
# ServiceのEndpoints（接続先Pod）を確認する
$ kubectl get endpoints -n book-app
```

### Observabilityの課題

現在、各Podのログは `kubectl logs` で個別に確認するしかない。複数サービスにまたがるリクエストで障害が発生した場合、どのサービスのどの処理で問題が起きたかを特定するのは困難である。メトリクスも収集されていないため、CPU使用率やレイテンシの傾向を把握できない。

### Service Meshの課題

サービス間の通信は平文のHTTPで行われている。クラスタ内の通信であっても、ゼロトラスト（Zero Trust）の考え方に基づけば暗号化すべきである。トラフィックの重み付けルーティングやリトライポリシーの設定もできない。

### Securityの課題

すべてのPodが他のすべてのPodと通信可能な状態である。NetworkPolicyによるアクセス制限が設定されていない。コンテナイメージに含まれる脆弱性のスキャンも行われておらず、ポリシー違反（特権コンテナの使用等）を検知する仕組みもない。

### CI/CD & GitOpsの課題

デプロイは `kubectl apply` による手動操作に依存している。Gitリポジトリのマニフェストとクラスタの実際の状態が乖離しても検知できない。新バージョンのデプロイで問題が発生した場合のロールバック手順も確立されていない。

### Platform Engineeringの課題

新しいマイクロサービスを追加する際、Deployment、Service、ConfigMap等のマニフェストを手作業で作成する必要がある。この手順は属人的であり、チームの知識として標準化されていない。インフラリソース（データベース、ストレージ等）の管理も宣言的に行われていない。
