# 第17章 Internal Developer Platform ― Backstage

Part 1〜4で、Observability、Service Mesh、Security、CI/CD & GitOpsの基盤を個別に構築してきた。しかし、新しいサービスを追加する際には、リポジトリの作成、マニフェストの用意、ArgoCD Applicationの登録、Observabilityの設定、NetworkPolicyの定義など、多くの手作業が必要である。Part 5では、これらの技術群を「開発者に提供するプラットフォーム」として統合するPlatform Engineeringを導入する。本章では、Backstageを用いてInternal Developer Platform（IDP）の基盤を構築する。

## 17.1 Platform Engineeringとは何か

### DevOpsの進化形

DevOpsは「You build it, you run it」の原則でサイロを取り払った。しかし、クラウドネイティブ技術の急速な発展により、開発者が理解すべき技術スタックは膨大になった。Kubernetes、Service Mesh、Observability、CI/CD、セキュリティポリシー ― これらすべてを各開発チームが個別に習得・運用するのは現実的ではない。

Platform Engineeringは、この認知負荷（Cognitive Load）の問題を解決するアプローチである。専門のプラットフォームチームがインフラと運用の複雑さを抽象化し、開発チームにセルフサービスのインターフェースとして提供する。

### IDPの三要素

IDPは以下の三要素で構成される。

1. **セルフサービス**: 開発者がチケットや依頼なしに、必要なリソースを自分で調達できる
2. **Golden Path**: 推奨される標準的な構成テンプレート。自由度を保ちつつ、ベストプラクティスに沿った設計を誘導する
3. **自動化**: テンプレートからの生成、CI/CDパイプライン、インフラプロビジョニングが自動で実行される

Team Topologiesの分類では、プラットフォームチームはストリームアラインドチーム（開発チーム）の認知負荷を減らすことを目的とするチームタイプである。プラットフォームチームの提供物はドキュメントやWikiではなく、「使えるプラットフォーム」そのものである。

### 認知負荷の三分類

Team Topologiesでは認知負荷を三つに分類している。

- **内在的認知負荷（Intrinsic）**: ビジネスドメインそのものの複雑さ。決済ロジック、在庫管理ルールなど。これは開発チームが本来担うべき負荷である
- **外在的認知負荷（Extraneous）**: 開発に直接関係しない技術的な複雑さ。Kubernetesマニフェストの記述、CI/CDパイプラインの設定、Observabilityの計装など。プラットフォームによって軽減すべき負荷である
- **学習的認知負荷（Germane）**: 新しい技術やドメインの学習に伴う負荷。適切な量であれば成長につながる

Platform Engineeringの目標は、外在的認知負荷を最小化し、開発チームが内在的認知負荷（ビジネスロジック）に集中できる環境を作ることにある。本書でPart 1〜4で構築した技術群は、そのままでは開発者にとって外在的認知負荷となる。Part 5では、これらをプラットフォームとして抽象化することで、外在的認知負荷を削減する。

## 17.2 Backstageの概要とアーキテクチャ

### Backstageとは

Backstageは、Spotifyが開発しCNCFに寄贈したオープンソースのInternal Developer Platform構築フレームワークである。2022年にCNCF Incubatingプロジェクトとなり、2025年にGraduatedプロジェクトに昇格した。

Backstageは「箱から出してすぐ使える完成品」ではなく、組織固有のIDPを構築するためのフレームワークである。プラグインシステムにより、CI/CD、Observability、クラウドサービスなど、任意のツールとの統合が可能である。

> 表17.1: Backstageのコア機能一覧

| コア機能 | 概要 | 用途 |
|---------|------|------|
| Service Catalog | サービス・API・インフラのカタログ管理 | サービスの発見、依存関係の把握、オーナーシップ管理 |
| Software Templates | テンプレートからの新規サービス生成 | Golden Pathの実装、標準化されたプロジェクト初期化 |
| TechDocs | ドキュメントのカタログ統合 | docs-as-codeによる技術ドキュメント管理 |
| Search | カタログ横断の全文検索 | サービス・API・ドキュメントの発見 |
| Kubernetes Plugin | クラスタリソースの可視化 | PodやDeploymentの状態をBackstage上で確認 |

Backstageは、フロントエンド（React SPA）とバックエンド（Node.js）で構成される。バックエンドはプラグインシステムを通じて外部サービスと連携する。カタログデータはPostgreSQLに格納される。

### プラグインエコシステム

Backstageの最大の強みはプラグインエコシステムにある。コア機能に加え、コミュニティが開発した多数のプラグインを組み合わせて、組織固有のIDPを構築する。

> 表17.1b: 代表的なBackstageプラグインの分類

| カテゴリ | プラグイン例 | 機能 |
|---------|-----------|------|
| CI/CD | GitHub Actions Plugin | ワークフロー実行状態の表示 |
| CI/CD | ArgoCD Plugin | ArgoCDアプリケーションの同期状態表示 |
| Monitoring | Grafana Plugin | Grafanaダッシュボードの埋め込み表示 |
| Monitoring | Prometheus Plugin | メトリクスのインライン表示 |
| Security | Snyk Plugin | 脆弱性スキャン結果の表示 |
| Cloud | AWS/GCP/OCI Plugin | クラウドリソースの可視化 |
| Documentation | TechDocs | Markdownドキュメントの自動レンダリング |
| Cost | Cost Insights Plugin | クラウドコストの可視化 |

プラグインはフロントエンドプラグインとバックエンドプラグインに大別される。フロントエンドプラグインはUI上にタブやカードを追加し、情報を可視化する。バックエンドプラグインはAPIエンドポイントを追加し、外部サービスとのデータ連携を担う。

プラグインの開発は、Backstage CLIの `@backstage/create-app` で生成されるスケルトンを起点に行う。TypeScriptとReactの知識があれば、組織固有のプラグインを開発できる。たとえば、OKEクラスタのリソース使用率を表示するカスタムプラグインや、社内のデプロイ承認フローを統合するプラグインなどが考えられる。

### Entityの概念

Backstageでは、管理対象のすべてのリソースをEntity（エンティティ）として表現する。Entityは `catalog-info.yaml` というYAMLファイルで定義され、Gitリポジトリのルートに配置する。Backstageは定期的にこのファイルを読み取り、カタログに反映する。

### カタログモデリングの設計

Backstageのカタログは、EntityのKindとそのリレーション（関係性）で構成される。組織の構造をカタログに正しくモデリングすることが、IDPの価値を最大化する鍵となる。

> 表17.1c: Entity Kindの一覧と用途

| Kind | 説明 | 用途例 |
|------|------|-------|
| Component | ソフトウェアコンポーネント | マイクロサービス、ライブラリ、Webサイト |
| API | 提供するAPI定義 | gRPC、REST、GraphQL、AsyncAPI |
| Resource | インフラリソース | データベース、キャッシュ、メッセージキュー |
| System | 関連Entityのグループ | マイクロサービス群をまとめた「注文システム」 |
| Domain | ビジネスドメイン | 「決済」「物流」「在庫管理」 |
| Group | チーム・組織 | 「バックエンドチーム」「SREチーム」 |
| User | 個人 | 開発者、管理者 |
| Location | カタログ定義ファイルの場所 | catalog-info.yamlのURL |

カタログモデリングでは、DDD（Domain-Driven Design）の境界づけられたコンテキスト（Bounded Context）を参考にDomainとSystemを設計するのが有効である。たとえば、ECサイトであれば「注文ドメイン」「商品ドメイン」「決済ドメイン」をDomainとして定義し、各ドメイン内のマイクロサービス群をSystemとしてグループ化する。

リレーションは `spec.dependsOn`、`spec.providesApis`、`spec.consumesApis`、`spec.system` などのフィールドで定義する。リレーションを正しく定義すると、Backstage UIの依存関係グラフで、サービス間の呼び出し関係やインフラ依存が可視化される。この可視化は、障害影響範囲の特定やアーキテクチャレビューに活用できる。

## 17.4 サービスカタログの構築

### catalog-info.yamlの記述

サンプルアプリケーションの各サービスをBackstageのカタログに登録する。

```yaml
# コード17.3: catalog-info.yaml（Componentの定義例）
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: order-service
  description: 注文管理サービス
  annotations:
    backstage.io/techdocs-ref: dir:.
    github.com/project-slug: your-org/order-service
    backstage.io/kubernetes-label-selector: app=order-service
  tags:
    - go
    - grpc
spec:
  type: service
  lifecycle: production
  owner: backend-team
  system: book-app
  providesApis:
    - order-api
  dependsOn:
    - resource:order-db
    - component:product-service
```

```yaml
# コード17.4: catalog-info.yaml（API定義とリレーション）
apiVersion: backstage.io/v1alpha1
kind: API
metadata:
  name: order-api
  description: 注文管理API
spec:
  type: grpc
  lifecycle: production
  owner: backend-team
  system: book-app
  definition: |
    syntax = "proto3";
    service OrderService {
      rpc CreateOrder(CreateOrderRequest) returns (Order);
      rpc GetOrder(GetOrderRequest) returns (Order);
      rpc ListOrders(ListOrdersRequest) returns (ListOrdersResponse);
    }
---
apiVersion: backstage.io/v1alpha1
kind: Resource
metadata:
  name: order-db
  description: 注文データベース
spec:
  type: database
  owner: backend-team
  system: book-app
```

> 表17.2: catalog-info.yamlの主要フィールド

| フィールド | 説明 | 例 |
|-----------|------|-----|
| apiVersion | Backstage APIバージョン | `backstage.io/v1alpha1` |
| kind | Entityの種類 | `Component`, `API`, `Resource`, `System`, `Group` |
| metadata.name | Entity名（一意） | `order-service` |
| metadata.annotations | Backstageプラグイン向けメタデータ | `backstage.io/kubernetes-label-selector` |
| spec.type | Entityのサブタイプ | `service`, `library`, `website` |
| spec.owner | 所有チーム/ユーザー | `backend-team` |
| spec.dependsOn | 依存するEntity | `component:product-service` |
| spec.providesApis | 提供するAPI | `order-api` |
| spec.consumesApis | 消費するAPI | `product-api` |

## 17.5 Software Templateによる新サービス作成

### Golden Pathの実装

Software Templateは、Backstageが提供する新規サービスの足場生成（Scaffolding）機能である。開発者はBackstage UIからテンプレートを選択し、パラメータを入力するだけで、標準構成のリポジトリが自動生成される。

図17.6: Software Templateのワークフロー

```mermaid
sequenceDiagram
    participant Dev as 開発者
    participant BS as Backstage UI
    participant Scaffolder as Scaffolder
    participant GH as GitHub
    participant Catalog as Service Catalog

    Dev->>BS: テンプレートを選択
    BS->>Dev: パラメータ入力フォーム
    Dev->>BS: パラメータ入力<br/>(サービス名、チーム等)
    BS->>Scaffolder: テンプレート実行
    Scaffolder->>GH: リポジトリ生成<br/>スケルトンコード配置
    Scaffolder->>GH: catalog-info.yaml配置
    GH-->>Catalog: 自動検出・登録
    Scaffolder->>Dev: 完了通知<br/>リポジトリURL
```

```yaml
# コード17.5: template.yaml（Software Template定義）
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: go-microservice
  title: Goマイクロサービス
  description: 標準構成のGoマイクロサービスを生成する
  tags:
    - go
    - microservice
    - recommended
spec:
  owner: platform-team
  type: service

  parameters:
    - title: サービス情報
      required:
        - serviceName
        - owner
        - description
      properties:
        serviceName:
          title: サービス名
          type: string
          pattern: "^[a-z][a-z0-9-]*$"
        owner:
          title: オーナーチーム
          type: string
          ui:field: OwnerPicker
        description:
          title: 説明
          type: string

    - title: インフラ要件
      properties:
        needsDatabase:
          title: データベースが必要
          type: boolean
          default: false
        needsCache:
          title: キャッシュが必要
          type: boolean
          default: false

  steps:
    - id: fetch-skeleton
      name: スケルトン生成
      action: fetch:template
      input:
        url: ./skeleton
        values:
          serviceName: ${{ parameters.serviceName }}
          owner: ${{ parameters.owner }}
          description: ${{ parameters.description }}

    - id: publish
      name: GitHubリポジトリ作成
      action: publish:github
      input:
        allowedHosts: ["github.com"]
        repoUrl: github.com?owner=your-org&repo=${{ parameters.serviceName }}
        description: ${{ parameters.description }}
        defaultBranch: main

    - id: register
      name: カタログ登録
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps.publish.output.repoContentsUrl }}
        catalogInfoPath: /catalog-info.yaml

  output:
    links:
      - title: リポジトリ
        url: ${{ steps.publish.output.remoteUrl }}
      - title: カタログページ
        url: /catalog/default/component/${{ parameters.serviceName }}
```

### テンプレートのスケルトン

テンプレートが生成するリポジトリには、サービスコード、Dockerfile、Kustomize構成、catalog-info.yamlが含まれる。

> 表17.3: テンプレートに含まれるファイル一覧

| ファイル | 役割 |
|---------|------|
| `cmd/server/main.go` | サービスエントリポイント |
| `Dockerfile` | マルチステージビルド用Dockerfile |
| `catalog-info.yaml` | Backstageカタログ登録用メタデータ |
| `base/kustomization.yaml` | Kustomize base構成 |
| `base/deployment.yaml` | Deploymentマニフェスト |
| `base/service.yaml` | Serviceマニフェスト |
| `overlays/dev/kustomization.yaml` | 開発環境オーバーレイ |
| `overlays/prod/kustomization.yaml` | 本番環境オーバーレイ |
| `.github/workflows/ci.yaml` | GitHub Actions CIワークフロー |

```go
// コード17.6: テンプレートのスケルトン（Goサービス）
// skeleton/cmd/server/main.go
package main

import (
	"log"
	"net/http"
	"os"
)

func main() {
	port := os.Getenv("PORT")
	if port == "" {
		port = "8080"
	}

	mux := http.NewServeMux()
	mux.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
	})
	mux.HandleFunc("/readyz", func(w http.ResponseWriter, r *http.Request) {
		w.WriteHeader(http.StatusOK)
	})

	log.Printf("${{ values.serviceName }} starting on port %s", port)
	if err := http.ListenAndServe(":"+port, mux); err != nil {
		log.Fatal(err)
	}
}
```

```yaml
commonLabels:
  app: ${{ values.serviceName }}
  team: ${{ values.owner }}
```

### テンプレートオーサリングの実践

Software Templateの開発では、以下の設計原則を意識する。

**パラメータ設計**: テンプレートのパラメータは、開発者が理解しやすい抽象度で設計する。Kubernetes固有の概念（ReplicaSet、PVC等）を直接パラメータ化するのではなく、「データベースが必要か」「サービスの規模はどの程度か」といったビジネス要件に近い抽象度で提供する。ui:fieldを活用して、OwnerPicker（チーム選択）やEntityPicker（既存Entity参照）などのリッチなフォームコントロールを提供できる。

**ステップの設計**: stepsセクションでは、Backstageが提供する組み込みアクション（Built-in Actions）を組み合わせる。代表的なアクションを以下に示す。

> 表17.3b: Software Template組み込みアクションの一覧

| アクション | 機能 | 用途 |
|-----------|------|------|
| fetch:template | テンプレートファイルのフェッチと変数展開 | スケルトンコードの生成 |
| fetch:plain | テンプレート展開なしのファイルコピー | 静的ファイルの配置 |
| publish:github | GitHubリポジトリの作成 | リポジトリの自動生成 |
| publish:github:pull-request | PRの作成 | 既存リポジトリへの変更提案 |
| catalog:register | カタログへの登録 | 生成したEntityの登録 |
| catalog:write | catalog-info.yamlの書き込み | カタログファイルの生成 |
| debug:log | デバッグログの出力 | テンプレート開発時のデバッグ |

組み込みアクションで不足する場合は、カスタムアクションを開発できる。たとえば、ArgoCD Applicationを自動登録するアクションや、Crossplane Claimを自動投入するアクションなどが考えられる。

**条件分岐**: テンプレートでは、パラメータの値に基づいてファイルの生成を制御できる。Nunjucks（テンプレートエンジン）の条件分岐を使い、`needsDatabase` が `true` の場合のみデータベース関連のマニフェストを生成する。スケルトン内のファイルで `{% if values.needsDatabase %}...{% endif %}` の構文を使用する。

### Golden Pathの設計思想

Golden Pathの設計で重要なのは、「自由度」と「標準化」のバランスである。

- **強制すべき項目**: ヘルスチェックエンドポイント、OTel計装、catalog-info.yaml、CIパイプライン
- **選択可能な項目**: データベースの有無、キャッシュの有無、レプリカ数
- **自由な項目**: ビジネスロジックの実装、内部のパッケージ構成

テンプレートはベストプラクティスへの入口であり、強制ではない。開発者はテンプレートで生成した後、必要に応じてカスタマイズできる。

### テンプレートのバージョニングと進化

テンプレートは一度作って終わりではなく、プラットフォームの進化とともにバージョンアップしていく。テンプレートの更新にあたって考慮すべき点を挙げる。

**既存サービスへの影響**: テンプレートを更新しても、既に生成済みのサービスには影響しない。テンプレートはリポジトリ生成時に一度だけ使用され、その後のリポジトリはテンプレートから独立して進化する。この性質は利点でもあり課題でもある。プラットフォーム全体のセキュリティパッチやベストプラクティスの更新を既存サービスに適用するには、別途PRを作成するアプローチ（publish:github:pull-requestアクション）が有効である。

**複数テンプレートの管理**: 組織が成長すると、言語やフレームワークごとに複数のテンプレートが必要になる。Goマイクロサービス、Pythonデータパイプライン、Reactフロントエンドなど、用途に応じたテンプレートを用意する。テンプレート間で共通の設定（CI/CD、Observability、Security）は、共有スケルトンとして切り出し、再利用する。

BackstageによるサービスカタログとSoftware Templateで、IDPの「入口」が構築された。しかし、テンプレートで生成されるのはGitHubリポジトリまでであり、サービスが必要とするデータベースやキャッシュなどのインフラは手動でプロビジョニングする必要がある。次章では、Crossplaneを導入し、Kubernetesマニフェストによるインフラのセルフサービスプロビジョニングを実現する。
