## 節の構成

### 11.1 サプライチェーンセキュリティの全体像

**概要**: ソフトウェアサプライチェーンにおける攻撃ベクトルを整理し、「ビルド → 保管 → デプロイ」の各段階で必要なセキュリティ対策を示す。

**図表**:
- 図11.1: コンテナサプライチェーンの攻撃面 ― ソースコード → ビルド → レジストリ → デプロイの各段階のリスク

**キーポイント**:
- サプライチェーン攻撃の実例と影響
- SLSA（Supply-chain Levels for Software Artifacts）フレームワークの概要
- 本章でカバーする範囲: 脆弱性スキャン（Trivy）、署名・検証（Sigstore/cosign）、デプロイ時の署名検証（Gatekeeper連携）

---

### 11.2 Trivy ― イメージ脆弱性スキャン

**概要**: Trivyの概要と、コンテナイメージ・Kubernetesマニフェスト・ファイルシステムに対する脆弱性スキャンの実践を行う。

**図表**:
- 図11.2: Trivyのスキャン対象 ― イメージ、ファイルシステム、Kubernetesクラスタ、IaC

**キーポイント**:
- Trivyの特徴: OSパッケージ + 言語パッケージの脆弱性検出、マニフェストの設定ミス検出
- 実践1: サンプルアプリのコンテナイメージに対する脆弱性スキャン（`trivy image`）
- 実践2: Kubernetesマニフェストに対する設定ミススキャン（`trivy config`）
- 脆弱性の重大度（CRITICAL / HIGH / MEDIUM / LOW）と対応方針
- `.trivyignore` による誤検知・許容リスクの除外

---

### 11.3 SBOM ― ソフトウェア部品表の生成と活用

**概要**: SBOM（Software Bill of Materials）の概念と、Trivyを使ったSBOM生成・脆弱性管理への活用を解説する。

**図表**:
- 図11.3: SBOMの位置づけ ― イメージ → SBOM生成 → 脆弱性DB照合 → レポート

**キーポイント**:
- SBOMとは: ソフトウェアに含まれるすべてのコンポーネントの一覧
- 標準フォーマット: SPDX、CycloneDX
- 実践: Trivyを使ったCycloneDX形式のSBOM生成（`trivy image --format cyclonedx`）
- SBOMを活用した継続的な脆弱性モニタリング

---

### 11.4 Sigstore / cosign ― イメージ署名と検証

**概要**: Sigstoreプロジェクトの概要と、cosignを使ったコンテナイメージへの署名・検証の実践を行う。

**図表**:
- 図11.4: cosignによる署名・検証フロー ― ビルド → cosign sign → レジストリ（イメージ + 署名）→ cosign verify → デプロイ

**キーポイント**:
- Sigstoreの構成要素: cosign（署名）、Rekor（透過性ログ）、Fulcio（証明書発行）
- キーペア方式 vs Keyless方式（OIDC認証）
- 実践: cosignによるイメージ署名とOCIRへのプッシュ
- 実践: cosignによる署名検証
- OKE環境でのOCIR連携のポイント

---

### 11.5 Gatekeeper連携 ― 署名なしイメージのデプロイ拒否

**概要**: Ch.10のGatekeeperとcosignの署名検証を連携させ、署名済みイメージのみデプロイ可能にするポリシーを実装する。

**図表**:
- 図11.5: 署名検証付きデプロイフロー ― イメージ → Gatekeeper Admission → cosign verify → 許可/拒否

**キーポイント**:
- Gatekeeperの外部データ機能またはcosign-gatekeeper-providerの活用
- ConstraintTemplate: イメージ署名の存在を検証するRegoポリシー
- 実践: 署名なしイメージのデプロイが拒否されることを確認
- 実践: 署名済みイメージのデプロイが成功することを確認
- Ch.10のGatekeeperが未導入の場合のスタンドアロン検証方法も補足

---

### 11.6 まとめと次章への橋渡し

**概要**: 本章の内容を振り返り、Ch.9〜11で構築した個別のセキュリティ機能を統合監査基盤として可視化する必要性を提示して第12章へつなげる。

**キーポイント**:
- 本章のまとめ: イメージの脆弱性スキャン、SBOM、署名・検証、デプロイ時の署名検証を実現した

- 次章予告: Falcoのランタイム検知を追加し、OPA違反・Trivyスキャン結果・ランタイムイベントをObservability基盤で統合してセキュリティダッシュボードを構築する

## 章末の理解度チェック

1. **サプライチェーンの攻撃面**: コンテナサプライチェーンにおける攻撃ベクトルを3つ挙げ、それぞれに対する対策を述べよ。
2. **Trivyのスキャン種別**: `trivy image` と `trivy config` の違いを説明し、それぞれどのようなリスクを検出するか述べよ。
3. **SBOMの意義**: SBOMを生成・管理する利点を、脆弱性管理の観点から2つ挙げよ。
4. **cosignの署名検証**: cosignのキーペア方式とKeyless方式の違いを説明し、CI/CD環境ではどちらが適しているか理由とともに述べよ。

## 章間の接続

### 前の章からの接続（Ch.10 → Ch.11）
- Ch.10のGatekeeperで「信頼レジストリ制限」ポリシーを実装したが、レジストリの信頼性だけではイメージの安全性は保証できない
- Ch.11ではイメージそのものの脆弱性スキャンと署名検証を導入し、より根本的なサプライチェーンセキュリティを実現する
- Ch.10のGatekeeper基盤を再利用し、署名検証ポリシーを追加する実践を含む

### 次の章への接続（Ch.11 → Ch.12）
- Ch.9（RBAC + NetworkPolicy）、Ch.10（OPA/Gatekeeper）、Ch.11（Trivy + Sigstore）の個別セキュリティ機能が出揃った
- Ch.12ではFalcoによるランタイム検知を追加し、すべてのセキュリティイベントをObservability基盤（Ch.5）と統合してセキュリティ監査ダッシュボードを構築する
- Ch.11のTrivyスキャン結果とGatekeeper違反情報はCh.12でメトリクスとして統合される
