### 目的

第2章〜第4章で個別に構築したメトリクス、ログ、トレースの3つのシグナルを、Grafanaを中心とした統合ダッシュボードに集約する。Three Pillarsの相関によって障害の検知から原因特定までを一気通貫で行える基盤を完成させる。SLI/SLOの設計と運用についても扱い、Observabilityを「ツールの寄せ集め」から「運用基盤」へ昇華させる。

### 前提知識

- 第2章: Prometheusによるメトリクス収集とPromQLの基本
- 第3章: Fluent Bit + Lokiによるログ集約とLogQLの基本
- 第4章: OpenTelemetry + Jaegerによる分散トレーシングとコンテキスト伝播
- Grafanaの基本操作（データソース追加、パネル作成）

## 節の構成

### 5.1 Three Pillarsの相関 ― なぜ統合が必要なのか

**概要**: メトリクス、ログ、トレースを個別に運用する場合の課題を明示する。障害発生時に3つのツールを行き来する非効率さを具体的なシナリオで示し、統合の必要性を動機づける。Three Pillarsの相関関係（メトリクスで異常検知、トレースで原因箇所特定、ログで詳細確認）を体系的に解説する。

**図表**:
- 図5.1: Three Pillarsの相関関係図（メトリクス・ログ・トレースの相互参照フロー）

**キーポイント**:
- メトリクスは「何が起きているか」、トレースは「どこで起きているか」、ログは「なぜ起きたか」を示す
- 3つのシグナルを相関させることで、検知から原因特定までの時間（MTTR）を短縮できる
- 相関の鍵はタイムスタンプ、ラベル（サービス名等）、トレースIDの3つ

**概要**: Grafanaに Prometheus、Loki、Jaegerの3つのデータソースを統合し、単一のダッシュボード上でThree Pillarsを横断的に参照できる環境を構築する。ダッシュボード設計の原則（USE/REDメソッド）とパネル構成のベストプラクティスを解説する。

**図表**:

- Grafanaのデータソース設定（Prometheus、Loki、Jaeger）を一元管理する
- REDメソッド（Rate、Errors、Duration）に基づくパネル配置
- テンプレート変数（namespace、service）による動的フィルタリング

### 5.3 Exemplars ― メトリクスからトレースへのドリルダウン

**概要**: PrometheusのExemplars機能を使い、メトリクスのデータポイントから対応するトレースへ直接ジャンプする仕組みを構築する。OpenTelemetryのSDKからExemplarsを付与する設定と、Grafana上でのドリルダウン操作を実践する。

**図表**:
- 図5.3: Exemplarsによるメトリクスからトレースへのドリルダウンフロー

**キーポイント**:
- ExemplarsはメトリクスデータポイントにトレースIDを紐づける仕組み
- OTelのSDK設定でExemplarsを有効化する手順
- GrafanaでメトリクスパネルのデータポイントからJaegerのトレースビューへ遷移する操作

---

### 5.4 SLI/SLOの設計と実装

**概要**: Observability基盤の上にSLI（Service Level Indicator）とSLO（Service Level Objective）を定義し、サービスの信頼性を定量的に管理する方法を解説する。サンプルアプリケーションに対して具体的なSLI/SLOを設計し、エラーバジェットの考え方を導入する。

**図表**:
- 表5.2: サンプルアプリケーションのSLI/SLO定義表
- 図5.4: エラーバジェットの消費状況を示すダッシュボードパネル

**キーポイント**:
- SLIはユーザー体験に直結する指標（可用性、レイテンシ、スループット）から選定する
- SLOは「99.9%のリクエストが200ms以内に応答する」のように具体的な数値で定義する
- エラーバジェット（Error Budget）はSLO違反の許容量であり、リリース判断の基準となる
- PromQLでSLI/SLOを計算するクエリの実装

**概要**: 意図的に障害を注入し、構築したObservability基盤が障害の検知・特定・解決にどのように貢献するかを検証する。メトリクスでの異常検知、トレースでのボトルネック特定、ログでの根本原因確認という一連のフローを実践する。

**図表**:

- 表5.3: 障害シナリオ一覧（レイテンシ増大、エラー率上昇、Pod再起動）

**キーポイント**:
- 障害注入の方法（サンプルアプリのレスポンス遅延、エラー率増加のエンドポイント活用）
- Grafanaのアラートルールによる異常検知の設定
- メトリクス → トレース → ログの順で原因を絞り込む実践的なトラブルシューティングフロー

## 章末の理解度チェック

1. Three Pillarsの各シグナル（メトリクス、ログ、トレース）が障害対応においてそれぞれどのような役割を果たすか説明せよ。
2. Exemplarsとは何か。メトリクスからトレースへのドリルダウンがどのように実現されるか説明せよ。
3. SLIとSLOの違いを説明し、サンプルアプリケーションの注文サービスに対して適切なSLI/SLOを1つ設計せよ。
4. エラーバジェットとは何か。エラーバジェットが枯渇した場合にどのような判断を行うべきか説明せよ。
5. 障害発生時にメトリクス → トレース → ログの順で原因を絞り込む手順を、具体的な操作を含めて説明せよ。

## 章間の接続

### 前の章からの接続（Ch.2〜4 → Ch.5）

第2章〜第4章では、メトリクス（Prometheus）、ログ（Fluent Bit + Loki）、トレース（OpenTelemetry + Jaeger）をそれぞれ個別に導入した。各ツールは独立して動作しているが、障害対応時にはツール間を手動で行き来する必要があり、効率的なトラブルシューティングができない状態にある。本章では、これら3つのシグナルをGrafanaで統合し、SLI/SLOに基づく運用基盤として完成させる。

### 次の章への接続（Ch.5 → Ch.6, Ch.7）

本章で構築したObservability基盤は、サンプルアプリケーションのアプリケーションレイヤーのテレメトリを収集している。しかし、サービス間のネットワーク通信に関する可視化・制御はまだ行えていない。Part 2では、Istio（第6章）またはCilium（第7章）によるService Meshを導入し、通信レイヤーの可視化と制御を実現する。第8章では、Service Meshが生成するテレメトリを本章のObservability基盤に統合する。
