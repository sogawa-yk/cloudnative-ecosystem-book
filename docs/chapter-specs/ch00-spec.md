## 節の構成

### 0.1 本書の対象読者と到達目標

**概要**: 本書が想定する読者像（Kubernetes基本リソースを運用した経験があるインフラエンジニア / SRE / プラットフォームエンジニア）を明示する。読了後に到達する5つの技術領域（Observability、Service Mesh、Security、CI/CD & GitOps、Platform Engineering）のスキルセットを一覧で示す。

**図表**:
- 図0.1: 読者のスキルマップ — 読む前と読んだ後の比較図（レーダーチャート形式のテキスト図）

**キーポイント**:
- 前提知識の明確化（Pod / Deployment / Service / ConfigMap / Ingress / Namespaceの理解、kubectl操作、YAML読み書き、Helm基本利用、Git基本操作）
- 前提としない知識の明確化（本書で扱う各ツール群）
- 到達目標の6項目を簡潔に列挙

### 0.2 書籍の構造 — 概念・実践・統合の3層モデル

**概要**: 本書の特徴的な構成パターン「概念 → 実践 → 統合」の3層モデルを解説する。各Partが個別ツールの学習で終わらず、統合章でツール間の組み合わせ方を学ぶという設計思想を伝える。Part 0〜Part 5の全体像を俯瞰する。

**図表**:
- 図0.2: 書籍の3層構造モデル — 概念・実践・統合の関係を示す図
- 図0.3: Part構成と章間依存関係の全体図（book-architecture.mdの簡略版）

**キーポイント**:
- 「積み上げ型」の構成: サンプルアプリに技術を段階的に乗せていく
- 各Partの冒頭で「今の課題は何か」を提示してから技術を導入する課題起点アプローチ
- 統合章の意義: 個別ツールの寄せ集めではなく統合基盤として機能させる

### 0.3 読書パス

**概要**: 読者の目的別に推奨する読書パスを提示する。全章通読のフルパスに加え、Observability特化、Security特化、CI/CD特化などの選択的パスを案内する。並列読書が可能な章の組み合わせも明示する。

**図表**:
- 図0.4: 読書パスマップ — 目的別の推奨パスをフローチャートで表現

**キーポイント**:
- フルパス（推奨）: Ch.0 → Ch.1 → Part 1 → Part 2 → Part 3 → Part 4 → Part 5
- Observability特化パス: Ch.0 → Ch.1 → Ch.2 → Ch.3 → Ch.4 → Ch.5
- Security特化パス: Ch.0 → Ch.1 → Ch.9〜11 → Ch.5 → Ch.12
- CI/CD特化パス: Ch.0 → Ch.1 → Ch.2 → Ch.11 → Ch.13〜16
- Part 1内のCh.2〜4は並列読書可能

### 0.4 実行環境の準備

**概要**: 本書のハンズオンを実行するための環境要件を説明する。メイン環境としてOKE（Oracle Container Engine for Kubernetes）を使用し、Kustomizeのoverlay方式で環境差分を吸収する戦略を紹介する。kind / minikubeなどの代替環境についても触れる。

**図表**:
- 図0.5: 環境構成の概要図 — OKE / kind / minikubeの環境差分とKustomize overlayの関係

**キーポイント**:
- メイン環境: OKE v1.34.1（3ノード、Oracle Linux 8.10、CRI-O）
- 代替環境: kind / minikube / kubeadm / 他マネージドK8s（EKS, GKE等）
- Kustomize base + overlays方式による環境差分の吸収（詳細は第1章）
- 必要なCLIツール一覧（kubectl, kustomize, helm, git等）

### 0.5 Namespace設計と本書の表記規則

**概要**: 書籍全体を通じて使用するNamespace設計（`book-`プレフィックス）と、本書内で使用するフォーマット規則（コードブロック、コマンド例、注意書きの表記方法）を説明する。

**図表**:
- 表0.1: Namespace一覧と対応Part

**キーポイント**:
- Namespace設計: book-app / book-observability / book-mesh / book-security / book-cicd / book-platform
- コード例の言語: Go（サンプルアプリ）、Python（ユーティリティ）、YAML（マニフェスト）
- 表記規則: コマンド例の `$` プレフィックス、出力例の表示方法、補足・警告のアイコン
