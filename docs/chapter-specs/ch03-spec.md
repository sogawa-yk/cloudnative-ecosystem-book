## 節の構成

### 3.1 なぜログ集約が必要か — サンプルアプリの課題から

**概要**: 第1章で確立したサンプルアプリケーションを起点に、ログが分散している状態の問題を具体的に示す。`kubectl logs`で1つずつPodのログを確認する非効率さ、Podの再起動でログが消失する問題、複数サービスにまたがるリクエストのログを追跡できない問題を提示する。

**図表**:
- 図3.1: 分散したログの課題 — kubectl logsで個別確認する現状 vs ログ集約後の統合検索

**キーポイント**:
- `kubectl logs`の限界: Pod単位でしかログを見られない、Podの再起動・削除でログが消失
- 複数サービスにまたがるリクエストのトレースが困難
- ログの検索・フィルタリングが手動かつ非効率
- ログ集約パイプラインによる解決の方向性

### 3.2 ログ収集パイプラインの設計

**概要**: ログ収集パイプラインの全体設計を解説する。収集（Collection）→ 加工（Processing）→ 転送（Routing）→ 保存（Storage）の各フェーズの役割と、Fluent Bit + Lokiの構成でそれぞれを担当するコンポーネントを説明する。

**図表**:
- 図3.2: ログ収集パイプラインの4フェーズ（Collection → Processing → Routing → Storage）
- 図3.3: Fluent Bit + Lokiのアーキテクチャ全体図

**キーポイント**:
- パイプラインの4フェーズモデル
- Fluent Bitの位置付け: 軽量なログ収集・加工エージェント（Fluentdとの比較）
- Lokiの位置付け: ログの索引をラベルのみに限定した軽量ログ集約システム（Elasticsearchとの比較）
- Lokiの設計思想: 「Like Prometheus, but for logs」— ラベルベースのインデックス
- DaemonSetパターンによるノード単位のログ収集

### 3.3 構造化ログの設計

**概要**: サンプルアプリケーションのログをJSON形式の構造化ログに変更する。構造化ログの利点（パース容易性、フィールドベースの検索、機械可読性）を説明し、Goアプリケーションでの実装方法を示す。

**図表**:
- 図3.4: 非構造化ログ vs 構造化ログの比較（検索・フィルタリングの観点）

**キーポイント**:
- 非構造化ログの問題点: 正規表現に頼るパース、フィールド抽出の不安定さ
- 構造化ログの設計原則: JSON形式、必須フィールド（timestamp, level, message, service, request_id）
- Go標準ライブラリ `log/slog` による構造化ログの実装
- ログレベル設計（DEBUG / INFO / WARN / ERROR）
- リクエストIDの付与（トレーシングとの連携の伏線）

### 3.4 Fluent Bitの導入 — DaemonSetでのデプロイ

**概要**: Fluent BitをDaemonSetとしてbook-observability Namespaceにデプロイし、各ノードのコンテナログを収集する設定を行う。Input / Parser / Filter / Outputの4つのセクションの役割を説明し、サンプルアプリのログを対象とした設定を構築する。

**図表**:
- 図3.5: Fluent Bitの処理パイプライン（Input → Parser → Filter → Output）
- 図3.6: DaemonSetによるノード単位のログ収集の概念図

**キーポイント**:
- Fluent Bit Helmチャートによるデプロイ
- DaemonSetのメリット: 各ノードに1つずつ配置、ノード上の全Podのログを収集
- Inputセクション: tail pluginで `/var/log/containers/*.log` を収集
- Parserセクション: JSON parserで構造化ログをパース
- Filterセクション: `kubernetes` filterでPodメタデータ（Namespace, Pod名, ラベル）を付与
- Outputセクション: Lokiへの転送設定
- book-app Namespaceのログのみを対象とするフィルタリング

### 3.5 Lokiの導入とログの検索

**概要**: LokiをHelmチャートでbook-observability Namespaceにデプロイし、Fluent Bitから転送されたログを保存・検索する。LogQLの基本クエリを学び、サンプルアプリのログに対して実用的な検索を実行する。

**図表**:
- 図3.7: Lokiのアーキテクチャ（Distributor / Ingester / Querier / Compactor）
- 表3.1: LogQLの基本構文一覧

**キーポイント**:
- Loki Helmチャートによるデプロイ（SingleBinary or Microservicesモード）
- Lokiのストレージ設計（チャンクストア、インデックスストア）
- LogQLの基本: ログストリームセレクタ `{namespace="book-app"}`
- フィルタ式: `|=`, `!=`, `|~`, `!~` による文字列フィルタリング
- パース式: `| json` による構造化ログのフィールド抽出
- 集約クエリ: `rate()`, `count_over_time()` によるログベースのメトリクス
- 実用クエリ例: エラーログの抽出、特定サービスのログ検索、リクエストIDによる横断検索

### 3.6 ログベースのアラートとベストプラクティス

**概要**: Lokiのルーラーを使ったログベースのアラート設定と、ログ運用のベストプラクティスを解説する。ログローテーション、保持期間の設計、コスト最適化の観点を含める。

**図表**:
- 表3.2: ログ運用のベストプラクティス一覧

**キーポイント**:
- Lokiのルーラーによるアラート設定（LogQLベースのアラートルール）
- ログローテーションとリテンション設計
- ログレベルによるフィルタリング（本番でDEBUGログを抑制する等）
- 機密情報のマスキング（Fluent Bitのfilterで対応）
- コスト最適化: 必要なログのみを保存、ラベルの適切な設計

## 章末の理解度チェック

1. Lokiが「Like Prometheus, but for logs」と呼ばれる理由を、インデックス戦略の観点から説明せよ
2. Fluent BitをDaemonSetでデプロイする利点を、Sidecarパターンと比較して説明せよ
3. 構造化ログ（JSON形式）が非構造化ログに対して持つ利点を3つ挙げよ
4. LogQLで `{namespace="book-app", app="order-service"} |= "error" | json | level="ERROR"` というクエリが何を検索しているか説明せよ
5. ログの保持期間を設計する際に考慮すべき要素を3つ挙げよ

## 章間の接続

### 前の章からの接続
- **第1章**: サンプルアプリケーションが`book-app` Namespaceで稼働している状態を前提とする。1.5節で明示した「ログがバラバラ」「障害時に何が起きたか分からない」の課題を本章で解決する。
- **第2章との関係**: 第2章（Prometheus）とは独立しており、並列で読むことが可能。ただしPart 1の統合章（第5章）では両方の知識が必要になる。

### 次の章への接続
- **第4章（Traces — OpenTelemetry + Jaeger）**: ログで「各サービスで何が起きたか」は把握できるようになったが、複数サービスにまたがるリクエストの因果関係を追跡するにはトレーシングが必要であることを示唆する。3.3節で導入したリクエストIDがトレーシングのtrace IDと連携する伏線を張る。
- **第5章（統合 — Observability基盤）**: 本章で導入したLokiのログをGrafanaダッシュボードに統合し、メトリクス・トレースと相関分析する。LogQLによるログベースのメトリクスをPrometheusのメトリクスと組み合わせる。
